What is FLive

Real-time football,
for every club

Amateur clubs play hundreds of matches a season with no coverage, no scoreboard, no way for supporters at home to follow along. FLive fixes that.

A reporter — a parent, a coach, anyone on the sideline — signs in and creates a match. During the game they log events from their phone: goals, cards, substitutions, corners, VAR reviews — whatever they track. Every event appears instantly on a public viewer feed.

Supporters anywhere in the world open a link — no app, no account — and watch the match unfold in real time. The feed refreshes automatically. When the match is over, a full summary is generated and stored permanently.

No camera, no stream, no bandwidth — just a text-based live ticker that works on any connection, including a patchy 3G signal on a remote pitch.

Features

Everything a live
ticker needs

From kick-off to full time — and penalties if it comes to that.

⚡

Real-time Event Feed

A single Firestore onSnapshot listener drives the viewer feed. Events appear within milliseconds of being logged. Finished matches automatically switch to a one-time fetch.

Firestore
📵

Offline Resilience

Reporters can keep logging events even when the pitch has no signal. Pending events queue in localStorage under flive_session and flush automatically on reconnect — no Firebase offline SDK required.

Offline-first
🎛️

Reporter Dashboard

Full match lifecycle management: schedule, edit, duplicate, cancel, and broadcast. Matches are grouped by Live / Upcoming / Finished with live pulse indicators and one-tap access to the control panel.

Reporter only
🗂️

Configurable Event Types

Each match has an enabledEventTypes[] array. Reporters choose from 22 gameplay events — goals, cards, substitutions, VAR, corners, shootout kicks — plus 8 fixed structural timeline events.

Per-match config
🥅

Extra Time & Penalties

Full match lifecycle: first half → second half → extra time → penalty shootout. Shootout tracked per kick with scored / missed / saved result. Scores maintained via Firestore increment().

Full lifecycle
🌍

Bilingual — CS / EN

Full i18n with next-intl. URL-based locale routing (/cs/…, /en/…). All UI strings in messages/cs.json and messages/en.json. Czech is the primary locale.

next-intl
🔒

Secure by Default

Firestore rules enforce isOwner() on every match write — update and delete require the authenticated caller to be the match's reporterId. Public reads are open; all writes are gated server-side.

Firestore Rules
📊

Post-match Summary

When a match finishes, writeMatchSummary() writes a compact summary object (scorers, cards, half-time score, shootout) directly onto the match document for instant load on the result page.

Auto-generated
📋

Multi-part Match Support

Configurable parts and partDuration per match — supports quarters, thirds, or any number of periods beyond the standard two halves. The control panel adapts automatically.

Flexible
📎

Share Links & QR Codes

Every match has a shareable viewer URL. Reporters can copy the link or display a QR code — generated with qrcode.react — directly from the dashboard so supporters can scan and follow along.

One tap
🌙

Dark / Light Theme

Full dark and light mode via next-themes. The design system is built on CSS custom properties in oklch color space — consistent contrast ratios across both themes without any opacity hacks.

oklch tokens
⚙️

Team & Player Management

Reporters manage their teams, squads, pitches and competitions in the Settings area. These are stored per-user in Firestore sub-collections and pre-populate the match creation form.

Reporter settings

Architecture

Key decisions
that matter

Non-obvious choices made deliberately — and the reasons behind them.

⚡ Single real-time listener

  • EventFeed.tsx is the only component using onSnapshot
  • All other data loading is one-time getDoc on mount
  • Finished matches automatically downgrade to a static fetch
  • Prevents runaway listener proliferation as the app grows

📵 Offline queue strategy

  • flive_session key in localStorage queues PendingEvent[]
  • Helpers: getBroadcastSession, setBroadcastSession, clearBroadcastSession
  • Events flush to Firestore automatically on reconnect
  • Firebase offline persistence intentionally not used — session queue is the strategy

🔐 Auth: React Context, not Zustand

  • AuthProvider wraps onAuthStateChanged → Firestore user lookup
  • Exposed via useAuth() → { user, loading, logout }
  • No Zustand anywhere in the project
  • AppShell guards protected routes; auth/layout.tsx guards auth pages

🛡️ Security at the rules layer

  • Firestore allow update, delete: if isOwner(resource.data)
  • isOwner() checks request.auth.uid == matchData.reporterId
  • Event sub-collection writes do a get() back to the parent match doc
  • Client-side guards are UX only — server rules are the real barrier

🎨 oklch color system

  • All tokens use full oklch(…) values — no bare HSL channels
  • Tailwind opacity modifiers (bg-primary/50) do not work — use color-mix() instead
  • Perceptually uniform — violet and amber stay vivid at the same lightness
  • Dark mode simply redefines the same custom properties

📂 Firestore data shape

  • users/{uid} — reporter profile keyed by Firebase Auth UID
  • matches/{matchId} — match doc with summary written on finish
  • matches/{matchId}/events/{eventId} — event sub-collection, never deleted
  • All reads/writes via firebaseServices.ts — no Firestore imports in components
