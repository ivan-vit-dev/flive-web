Architecture Overview

Read this before touching the code. These constraints are non-negotiable.

Auth — two layers

Server: src/proxy.ts validates HTTP-only session cookie via next-firebase-auth-edge. Redirects unauthenticated requests to login. Client: AuthProvider context wraps onAuthStateChanged → Firestore users/{uid} → useAuth().

Firestore access — one file

All reads and writes go through src/lib/firebaseServices.ts. Components never import db directly. Every document has createdAt, updatedAt (Timestamps) and reporterId/userId ownership fields.

Real-time — one listener

EventFeed.tsx is the only component that uses onSnapshot. All other data loading is one-time getDocs/getDoc on mount. EventFeed auto-switches to a one-time fetch when a match is finished. Do not add onSnapshot calls elsewhere.

Offline — localStorage queue

The flive_session localStorage key holds PendingEvent[] when signal drops. Helpers: getBroadcastSession, setBroadcastSession, clearBroadcastSession, saveMatchClock, clearMatchClock — all in src/lib/utils.ts. Never use Firebase offline persistence.

Events subcollection — never delete

matches/{matchId}/events/{eventId} is the permanent source of truth. writeMatchSummary() writes a compact summary field to the match doc for fast viewer access — but the events remain forever.

No Zustand — React Context only

Auth state lives in AuthProvider via useState. No Zustand anywhere. The useAuth() hook returns { user, loading, setUser, logout }. After registration, call setUser(userDoc) before router.push to prevent a race with middleware cookie validation.

Match status enum

type MatchStatus =
  "scheduled"          # Not yet started
  | "live_first"       # First half / first part running
  | "half_time"        # Between first and second half
  | "live_second"      # Second half running
  | "live_part"        # Part N running (multi-part mode)
  | "break"            # Between parts (multi-part mode)
  | "extra_time"       # Extra time running (continuation past full-time)
  | "penalty_shootout" # Shootout in progress
  | "finished"         # Final whistle blown, summary written
  | "cancelled"        # Match abandoned
  | "postponed"        # Match rescheduled

forwardRef requirement for Base UI wrappers

Button and Input wrap @base-ui/react primitives. Any wrapper used with react-hook-form's register() or Radix's asChild must use React.forwardRef and forward the ref to the primitive. This is already done for Input. Apply the same pattern to any new wrappers.

Radix asChild is incompatible with Base UI Button

DropdownMenuTrigger asChild and similar Radix patterns silently fail when the child is the flive-web Button component. Style the trigger directly via className instead of using asChild.

Auth Flow

Login / Register sequence

1. Firebase signInWithEmailAndPassword / createUserWithEmailAndPassword
2. cred.user.getIdToken()  →  idToken
3. fetch("/api/auth/login", { method: "POST", headers: { Authorization: `Bearer ${idToken}` } })
   → authMiddleware in proxy.ts intercepts, validates token, sets HTTP-only "session" cookie
4. getUserDoc(uid)  →  userDoc
5. setUser(userDoc)      # BEFORE router.push — prevents race condition
6. router.push(`/${locale}/dashboard`)

Logout sequence

1. fetch("/api/auth/logout", { method: "POST" })
   → authMiddleware clears "session" cookie
2. signOut(auth)          # Firebase client signout
3. setUser(null)

Request protection (proxy.ts)

Every request → authMiddleware checks "session" cookie:
  Valid token   → handleValidToken:
    - If /[locale]/auth/* path → redirect to /dashboard
    - Otherwise → pass through intl middleware
  Invalid token → handleInvalidToken:
    - If /[locale]/dashboard/* or /[locale]/broadcast/* → redirectToLogin
    - Otherwise → pass through intl middleware
  Error         → pass through intl middleware

Real-Time Pattern

Clock synchronisation

The reporter's clock advances via setInterval every 1 second — purely client-side. Every 30 seconds, updateMatchClock(matchId, seconds) writes two fields to Firestore:

currentMinute:   Math.floor(seconds / 60)
currentMinuteAt: serverTimestamp()   # when that minute *started*

Viewers compute elapsed time:

const elapsed = (Date.now() - match.currentMinuteAt.toDate().getTime()) / 1000;
const displaySeconds = match.currentMinute * 60 + elapsed;

This means all viewers see the same clock without polling — they just do local arithmetic against the stored timestamp.

EventFeed listener lifecycle

mount:
  if match.status === "finished" → getMatchEvents(id)   # one-time fetch
  else → onSnapshot(collection, query)                   # real-time

inside snapshot callback:
  if updated match.status === "finished" → unsubscribe and re-fetch once
  if score changed → call onMatchUpdate(updatedMatch)    # reporter scoreboard sync

unmount: unsubscribe()

Offline Resilience

Session queue structure

// localStorage key: "flive_session"
interface BroadcastSession {
  matchId: string;
  pendingEvents: PendingEvent[];
  clock?: {
    seconds: number;   # reporter's clock at time of save
    savedAt: number;   # Date.now() in ms
  };
}

interface PendingEvent {
  localId: string;          # client-generated UUID for dedup
  event: Omit<MatchEvent, "id" | "createdAt" | "updatedAt">;
  queuedAt: string;         # ISO timestamp string
}

Reconnect flush

When LiveControlPanel detects connectivity restored, it reads getBroadcastSession() and iterates pendingEvents, calling addMatchEvent for each. Successfully flushed events are removed from the session. On match finish, clearBroadcastSession() is called.

Clock restoration

On page reload, LiveControlPanel checks getBroadcastSession().clock. If present: it computes elapsed time since savedAt and resumes the clock from seconds + elapsed. This keeps the reporter's local clock consistent across page refreshes even without a network round-trip.

Patterns & Gotchas

proxy.ts — not middleware.ts Next.js 16 renamed the middleware entry to src/proxy.ts. The exported function must be named proxy. If you create a middleware.ts, the build will error. Do not create both files.
setUser before router.push on auth After login or registration, always call setUser(userDoc) before router.push. The dashboard middleware sees a valid cookie but AppShell reads auth from context — if context is empty, it may flash a redirect loop.
Opacity modifiers don't work with oklch tokens CSS custom properties hold full oklch(...) values. Tailwind's bg-primary/50 syntax produces transparent results. Use color-mix(in oklch, var(--primary) 50%, transparent) in raw CSS, or pick a different token.
No onSnapshot outside EventFeed EventFeed.tsx is the single authorised place for real-time Firestore listeners. All other data loading is one-time on mount. Adding another onSnapshot elsewhere will accumulate open listeners and cause memory leaks.
Radix asChild incompatible with Base UI Button Base UI doesn't expose a ref-compatible interface with Radix's asChild. Style DropdownMenuTrigger directly via className — never nest the flive-web <Button> inside it.
Score increments are atomic — addMatchEvent only Never manually patch homeScore or awayScore on a match doc. addMatchEvent uses Firestore increment() in the same write as the event. deleteMatchEvent does the reverse. Calling both independently risks double-counting.
Never delete the events subcollection The events subcollection is permanent. On match finish, writeMatchSummary() compiles a compact summary object onto the match doc — that summary is for fast viewer access only. The events remain the source of truth for any future reporting or auditing.
No Firebase offline persistence — use localStorage queue Firebase SDK offline persistence is explicitly not used. The flive_session localStorage queue is the offline strategy. It's simpler, fully controlled, and doesn't conflict with the server-side session cookie pattern.
