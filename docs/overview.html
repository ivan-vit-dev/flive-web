A live match broadcaster for amateur and youth football clubs — no TV budget, no production crew. One reporter, one phone, and every fan follows along in real time.

About

What is FLive?

FLive turns any club volunteer into a live match commentator. A reporter logs events in a browser dashboard; fans watch an auto-updating public feed on any device — no app install required.

Target audience

Czech and Slovak amateur and youth football clubs that can't afford a TV broadcast crew. Parents, fans, and club members who can't attend every away match and want a live feed.

Core concept

A reporter logs structured match events (goals, cards, substitutions, phases) from a dashboard. Viewers watch a real-time event feed at a shareable public URL. No accounts needed to watch.

Offline resilience

If the reporter loses mobile signal, events queue locally in localStorage. When connectivity returns, everything syncs automatically — the broadcast never misses a goal.

Club-ready

Reporters pre-configure their club's teams, players, pitches, and competitions in Settings. A saved team auto-fills the match creation form in seconds — ready for kick-off.

User Roles

Two user types, zero friction

Reporters need an account. Viewers need nothing — just a link.

📋

Reporter

An authenticated club volunteer who manages and broadcasts matches.

  • Create and configure matches (teams, players, event types, parts)
  • Start / pause / end match phases from the live control panel
  • Log all events in real time: goals, cards, subs, corners, VAR…
  • Keyboard shortcuts for the most common events
  • Copy a broadcast link and QR code to share with fans
  • Manage saved teams, players, pitches and competitions in Settings
  • Works offline — events queue and sync on reconnect
👁️

Viewer

An anonymous fan or parent who watches a live or finished match.

  • Browse all live and upcoming public matches on the home page
  • Watch a live event feed that auto-updates in real time
  • See the scoreboard, live clock, current match phase
  • Opt in to browser push notifications for goals and red cards
  • Share the broadcast link or QR code with others
  • Read the full event timeline of finished matches
Match Lifecycle

How a broadcast works

From creation to the final whistle — every step in order.

1

Create the match

Reporter fills the match form: home/away teams, venue, competition, scheduled time, number of parts, part duration, player list, and which event types to enable. A team picker auto-fills all of this from saved club data.

2

Share the broadcast link

The match instantly appears on the public home page. Reporter copies the broadcast URL or QR code and shares it with fans. Viewers can follow along even before kick-off.

3

Start the match

Reporter opens the dashboard control panel and clicks "Start match". A match_start structural event is logged, status changes to live_first, and the live clock begins. Viewers' feeds update immediately.

4

Log events in real time

Reporter taps event buttons (or uses keyboard shortcuts: G = goal, Y = yellow card…). Complex events open an EventForm dialog for player name, assist, added time, description. Simple events fire immediately.

5

Progress through phases

Control panel buttons advance the match: half-time → second half → full-time. Optional extra time and penalty shootout phases are one-click transitions. Each logs a structural TIMELINE_EVENT to mark the phase boundary permanently.

6

Finish and summarise

Reporter clicks "Finish match". writeMatchSummary() compiles scorers, cards, half-time score, extra-time flag, and shootout result directly onto the match document. The events subcollection is never deleted — it remains the source of truth.

Features

Main capabilities

Built specifically for the amateur match-day workflow — field-tested constraints, no bloat.

⚡

Real-Time Event Feed

Viewers see events the moment the reporter logs them — onSnapshot listener on the events subcollection, scoped to only the one component that needs it. Automatically switches to one-time fetch for finished matches.

📡

Live Match Clock

Reporter's clock advances client-side every second. Every 30 s it syncs to Firestore via currentMinuteAt timestamp. Viewers compute elapsed time client-side — all fans see the exact same clock without polling.

🗂️

Configurable Event Types

30 event types (8 structural + 22 gameplay). Reporters choose which gameplay events to enable per match — a youth U10 game won't need VAR reviews. Structural events (kick-off, half-time, etc.) are always present.

📴

Offline Queue

Events queue in localStorage under flive_session when the reporter's signal drops. Reconnect → auto-sync. The clock position is also cached so the timer survives a page refresh.

⚽

Score Auto-Management

Goals, own-goals, penalty scores, and penalty kicks all auto-increment/decrement the match score via Firestore increment() in the same write as the event — atomic, no double-counting.

🔫

Penalty Shootout

Full shootout tracking per kick: scored / missed / saved. Home and away shootout scores are maintained separately on the match document. Works seamlessly after extra time.

⌨️

Keyboard Shortcuts

Common events triggered from the keyboard — G=goal, O=own goal, Y=yellow card, R=red card and more. Designed for reporters holding a phone with one hand and keeping their eyes on the pitch.

📱

PWA — Install on Any Device

Service worker caches static assets for offline resilience. manifest.json enables "Add to Home Screen" on Android and iOS. Reporters can launch the dashboard without opening a browser.

🔔

Push Notifications for Viewers

Viewers can opt in to browser push notifications. Goals and red cards fire a notification in the background — fans at work still know what's happening.

🔗

Shareable Public Feed

Every match has a public URL (no login needed to view) and a QR code. Share it in a club WhatsApp group before kick-off and the feed is live for everyone in seconds.

🏆

Extra Time & Multi-Part Matches

Supports 1–4 configurable parts with custom durations. Extra time is a seamless continuation past the final whistle, not a separate phase. Matches with up to 4 parts work for indoor or festival football.

📊

Post-Match Summary

On finish, a compact summary object is written directly onto the match document — scorers, cards, half-time score, extra time flag, shootout result — so viewers can get the quick recap without querying the full events subcollection.

Architecture

Key decisions

Constraints that every developer must know before touching the codebase.

  • Two-layer auth — server cookie + client context Server layer: next-firebase-auth-edge validates an HTTP-only session cookie on every request via src/proxy.ts. Client layer: AuthProvider wraps onAuthStateChanged, exposes useAuth(). Login POSTs idToken to /api/auth/login; logout calls /api/auth/logout before Firebase signOut.
  • Single real-time listener — EventFeed only EventFeed.tsx is the only component that may use onSnapshot. All other data loading is one-time getDocs/getDoc on mount. For finished matches, EventFeed automatically switches to a one-time fetch. Never add onSnapshot elsewhere.
  • All Firestore access via firebaseServices.ts Components, hooks, and contexts never import db directly. Every read and write goes through src/lib/firebaseServices.ts. This keeps data logic testable and auditable in one place.
  • Offline resilience via localStorage session queue The flive_session key stores PendingEvent[] when the reporter loses signal. Helpers in src/lib/utils.ts: getBroadcastSession, setBroadcastSession, clearBroadcastSession, saveMatchClock. Never use Firebase's offline persistence — the session queue is the strategy.
  • Events subcollection is never deleted The matches/{matchId}/events/{eventId} subcollection is the permanent source of truth. writeMatchSummary() adds a compact summary object to the match doc for fast viewer access — but the events remain. Never delete them.
  • Next.js 16 middleware file is proxy.ts Next.js 16 renamed the middleware entry from middleware.ts to proxy.ts. The exported function must be named proxy. Both next-intl locale routing and auth session guard are chained inside this single file. Do not create a middleware.ts.
  • Tailwind oklch tokens — opacity modifiers disabled CSS custom properties use full oklch(...) values, so Tailwind's opacity modifier syntax (bg-primary/50) produces transparent results. Use color-mix(in oklch, var(--token) 50%, transparent) in raw CSS instead, or choose a different token.
  • No Zustand — auth is React Context only Auth state lives in AuthProvider via useState internally, exposed via useAuth(). After registration, call setUser(userDoc) before router.push to avoid the race where the middleware sees a valid cookie but the context hasn't populated yet.
